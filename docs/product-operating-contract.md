# Product Operating Contract

## Purpose of this document

This document converts the current product vision into operational rules for all future implementation work.

It exists to ensure that the project is built with the discipline of a company product, not the improvisation of a personal website.

Before any feature is designed, coded, or shipped, it must align with this contract.

## 1. Primary user

The primary audience is not the general public. It is composed of professional decision-makers who can create real opportunities for Andres.

The core user groups are:

- Restaurant owners
- Chefs
- Hotel hiring managers
- Recruiters

These users are evaluating credibility, standards, fit, and professional readiness. The product must help them make that judgment quickly and confidently.

## 2. Single conversion goal

The website has one primary conversion goal:

**Make an employer want to contact Andres for an interview.**

Every future page, section, interaction, and content decision must support that outcome directly or indirectly.

If a feature does not strengthen employer confidence or increase contact intent, it should not be prioritized.

## 3. Trust ladder

The product must guide users through a precise trust sequence:

### Attention

The first screen must immediately communicate that Andres is a serious professional, not a generic chef profile.

### Credibility

The experience must quickly signal standards, professionalism, and real-world readiness.

### Proof

The site must provide tangible evidence through experience, skills, curated visuals, and structured credentials.

### Human connection

The user must understand who Andres is as a person, how he works, and why he would be valuable in a serious team.

### Contact intent

The experience must reduce friction and make it easy for the right employer to take the next step.

This sequence is non-optional. Future implementation should be evaluated by whether it strengthens one or more levels of the trust ladder without weakening the earlier ones.

## 4. Page hierarchy

The first version of the product must be a **single-page landing experience**.

This is an intentional MVP constraint.

Reasons:

- The product needs one strong, focused trust journey before expanding into multiple routes.
- A single page reduces fragmentation during the first implementation cycle.
- It creates pressure to prioritize only the content that truly affects conversion.

Future pages may exist later, but the MVP must focus on one highly effective homepage.

## 5. MVP sections

The MVP homepage must include the following sections in a trust-building sequence:

### Hero

Create immediate positioning, clarity, and first-contact confidence.

### Professional promise

State what kind of professional Andres is, what standards he represents, and why he deserves attention.

### Experience proof

Show career credibility through structured proof, not inflated claims.

### Skills

Present meaningful capabilities that help employers evaluate readiness and fit.

### Story

Introduce the human layer behind the professional profile. This section should create connection without losing seriousness.

### Gallery

Provide curated visual proof that supports standards, discipline, and taste.

This section is optional until real professional assets exist.

It must never appear as filler, placeholders, or empty visual structure.

### Resume download

Offer a clear path to deeper evaluation materials for serious employers.

### Contact CTA

Create a direct, low-friction invitation to initiate conversation or interview contact.

No additional MVP section should be added unless it clearly improves the conversion path.

## 6. Non-negotiable rules

The following rules apply to all future implementation work:

- No generic chef website language.
- No fake claims.
- No immigration-first messaging.
- No overdesigned effects.
- No slow loading.
- No section without a conversion purpose.
- Mobile must feel premium.
- English first.

These are product constraints, not stylistic suggestions.

## 7. Immigration messaging rule

Public messaging about relocation must remain professional and opportunity-oriented.

The public website should say:

> Available for international relocation.

The public website should not say:

> I do not have papers.

Immigration details must be handled privately after real employer interest exists.

The public experience should protect dignity, focus, and trust.

## 8. Analytics intent

Every important user action must be measurable because the product is meant to learn, not just exist.

The MVP analytics model should be designed to measure:

- Page visit
- Resume download
- WhatsApp click
- Email click
- LinkedIn click
- Scroll depth
- Return visit

These events matter because they reveal discovery, trust progression, and professional intent.

## 9. Definition of done for MVP

The MVP is done when a restaurant owner can open the page on mobile, understand Andres in under 30 seconds, trust his experience, download his resume, and contact him.

Anything less means the MVP is not truly complete, even if the page is visually attractive or technically functional.

## 10. Engineering behavior

Before implementing any feature, future Codex tasks must check whether the feature supports the conversion goal.

This check should happen before:

- Building UI
- Adding sections
- Introducing interactions
- Tracking new analytics events
- Expanding content structure
- Creating new pages

The default engineering question must be:

**Does this make the right employer more likely to trust Andres and contact him for an interview?**

If the answer is unclear, the feature should be reconsidered before implementation.

## Operating summary

This project is not building a personal website in the usual sense.

It is building a focused professional trust system with one public objective:

Make the right employer want to contact Andres.

That objective should govern product strategy, content decisions, technical architecture, future analytics, and all implementation work from this point forward.
