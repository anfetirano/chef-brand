# Engineering Constitution

## Purpose

This document is the permanent engineering constitution of this repository.

It defines how the product must be built, how engineering decisions must be evaluated, and how future Codex iterations must behave.

It is not optional guidance.

Every future implementation must respect it.

## Section 1: Engineering Philosophy

The goal is not to build pages.

The goal is to increase employer trust.

Every commit must improve at least one of these five outcomes:

- Trust
- Clarity
- Performance
- Conversion
- Maintainability

Nothing else should justify implementation work.

If a change does not improve one of these outcomes, it does not belong in the product.

## Section 2: Product Thinking

Every feature must answer one question before it is accepted:

**Why does this increase the probability of getting hired?**

If the feature cannot answer that question clearly, it should not exist.

The product is not a canvas for ideas, trends, or experiments without purpose. It is a professional trust system designed to create real opportunity.

## Section 3: Design Philosophy

The design standard of this repository is:

- Premium
- Minimal
- Elegant
- Calm
- Luxury hotel

The product should not feel like:

- A startup
- An agency website
- A portfolio template

The intended reference mindset is:

- Apple
- Aesop
- Four Seasons
- Michelin Guide

This means restraint, control, confidence, taste, and refinement. The visual experience should feel expensive because it is disciplined, not because it is overloaded.

## Section 4: Performance Philosophy

The product must be:

- Fast
- Lightweight
- SEO-first
- Accessibility-first

Engineering should minimize unnecessary JavaScript and avoid technical decisions that create weight without meaningful business value.

Animations must support storytelling.

There must never be animation for decoration alone.

Performance is not a technical nice-to-have. It is part of trust, discoverability, and conversion.

## Section 5: Content Philosophy

The content hierarchy of meaning is:

- Story before experience
- Experience before skills
- Skills before technology
- Human before resume
- Authenticity before marketing

Engineering decisions that structure or present content must protect this order.

The product should reveal who Andres is before trying to prove what tools or techniques he knows.

## Section 6: Analytics Philosophy

Nothing is tracked without purpose.

Every metric must answer a business question.

Analytics exists to understand whether the product is attracting qualified attention, building trust, and increasing contact intent.

If a metric cannot support a product, content, or business decision, it should not be implemented.

## Section 7: Coding Philosophy

All implementation work should follow these principles:

- Readable over clever
- Simple over complex
- Composition over duplication
- Scalable over shortcuts
- Document important decisions

Code should be understandable by future collaborators and easy to evolve without breaking the product's strategic direction.

## Section 8: AI Collaboration Rules

Future Codex iterations must:

- Question bad ideas
- Protect the product vision
- Reject unnecessary complexity
- Suggest improvements
- Never blindly execute prompts
- Behave like a senior Product Engineer

Codex is not expected to act like a passive generator.

It is expected to act like a responsible engineering partner with product judgment.

If a request weakens the product, creates noise, or introduces avoidable complexity, Codex should say so clearly and propose a better path.

## Section 9: Definition of Excellence

The product reaches excellence when a visitor feels:

> I already trust this chef.

before reaching the contact section.

This is the emotional threshold the experience must create.

Everything before contact should be designed to earn that feeling.

## Section 10: North Star

The objective is not to impress developers.

The objective is to make restaurant owners contact Andres.

Everything exists to maximize that outcome.

All future engineering, content structure, interface decisions, performance work, and analytics planning should be evaluated against this north star.

## Closing Principle

This repository is building a product, not a personal site in the conventional sense.

It should be engineered with discipline, editorial judgment, technical restraint, and strategic clarity.

When a future decision creates tension between aesthetics and trust, novelty and clarity, or complexity and conversion, the constitution is the tie-breaker.
